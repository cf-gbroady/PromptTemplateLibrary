Semantic Understanding

Why context and boundaries matter — the model fills gaps based on what seems likely, not what you actually meant.

How the model "understands" language

Tokens → Embeddings → Meaning

Step 1 — Tokenisation: text is broken into units

Iheardadogbarkloudlyatacat

Step 2 — Embeddings: tokens become vectors that encode meaning

WordEmbedding (simplified)Relationship
dog[10, 3, 2]—
cat[10, 3, 1]Very similar to dog
puppy[5, 2, 1]Related to dog
skateboard[-3, 3, 2]Far from dog
Talk track: "The model doesn't understand words the way a person does. Related words sit closer together in meaning-space — which is why the model connects 'dog,' 'puppy,' and 'cat' more naturally than 'dog' and 'skateboard.' This is also why the model will fill in gaps based on what seems likely unless you define the job clearly."
Semantic vs. keyword search

Why users don't need perfect keywords

User asks: "How do I make this page easier for students to understand?"

KEYWORD THINKING

Looks for exact words: page, students, understand — may miss useful sources

SEMANTIC THINKING

Infers related ideas: readability, plain language, accessibility, audience, clarity

Replicate the Human Process

Do not start with the agent screen. Start with the human workflow. If you can't describe what a good human helper would do, you probably can't instruct an agent well.

Human Process Map Template
Human StepWhat a human would checkAgent translation
1. IntakeWhat does the user want? What is missing?Clarifying questions and required inputs
2. TriageIs this in scope? Is it safe?Scope rules and guardrails
3. Gather contextWhat sources, files, or constraints matter?Web Search, Code Interpreter, user-provided files
4. Reason throughWhat steps would the human follow?Ordered workflow instructions
5. Produce outputWhat does a useful answer look like?Format, tone, length, citations, next steps
6. Review & escalateWhat requires human judgment?Caveats, confidence language, escalation instructions
Mini prompt: "If a knowledgeable human did this task well, what would they do first, second, third — and what would they refuse to do?"
Demo idea: Website Accessibility Review Assistant

Why this works: Relevant to web professionals, uses web search for public guidance, uses Code Interpreter for exported page data, requires human review, and illustrates semantic understanding naturally.

Build Basics — The 5-Step Model

Use this as your "plain English" explanation during the demo.

Step 1

Determine the use case

Define the task, user, information, tone, and success metric. Use PACE: Problem, Action, Context, Expectations.

Step 2

Create system instructions

Tell the agent what it is, who it helps, what process it follows, what tools it should use, how it should respond, and what it should not do.

Step 3

Add tools and sources

In AI Studio: Web Search and Code Interpreter. Identify trusted sources and describe why each source is relevant.

Step 4

Create starter prompts

One easy, one realistic, one that tests whether the agent knows when to use a tool.

Step 5

Test and iterate

Test for accuracy, comprehension, tone, style, behavior, output, and guardrails. Change one thing at a time. Try to break it before sharing it.

Capability distinction: "Today we are building within a lightweight AI Studio experience that has two tools: Web Search and Code Interpreter. Official nebulaONE agents can be configured more deeply in the full environment, but the design discipline is the same: define the audience, outcome, information sources, process, guardrails, starter prompts, and tests before you build."

Section 8 — First System Instruction Draft

You are [agent role] for [audience]. Your job is to help users [primary outcome]. Replicate this human process: 1. First, [intake / clarify]. 2. Then, [gather or inspect context]. 3. Then, [analyze / compare / draft / review]. 4. Finally, [produce the expected output and next steps]. When responding: - Ask clarifying questions if the request is missing [key inputs]. - Use web search when the answer depends on current or source-specific information. - Use code interpreter when the user provides data/files or asks for calculations, charts, transformations, or structured analysis. - Explain your reasoning briefly and practically. - Format responses as [preferred format]. - Use a [tone] tone. Context rules: - Treat [source/tool/context] as authoritative for [specific situation]. - If required context is missing, ask for it before giving a final answer. Guardrails: - Do not invent facts, policies, URLs, data, or institutional requirements. - If sources conflict or are missing, say so and recommend what to verify. - Do not make high-stakes decisions for the user; provide options and recommended next steps for human review. - Do not request or expose sensitive information unless the environment and policy explicitly allow it.

Break-It Testing

Testing is part of design. Make participants test the agent with bad prompts before they trust it.

Break-It Checklist
  • ☐ A vague prompt
  • ☐ A prompt outside its purpose
  • ☐ A request that needs current information
  • ☐ A request that should trigger Code Interpreter
  • ☐ A request that should not be answered without human review
  • ☐ A request that tempts it to invent a policy, source, or fact
  • ☐ A request from a user with a different skill level than expected
  • ☐ A request that skips a key human-process step
  • ☐ A request that gives conflicting information
  • ☐ A request that includes sensitive information that should not be used with web search
Success Metric

Pick one concrete success metric before sharing:

  • ☐ Saves users at least ______ minutes per task
  • ☐ Reduces repeated questions about ______
  • ☐ Improves consistency of ______
  • ☐ Helps users produce a usable first draft of ______
  • ☐ Helps users analyze or transform ______
  • ☐ Helps users follow a consistent human process for ______
  • ☐ Other: ______
Facilitator reminder: Model behavior is not deterministic. Language models cannot be relied on to behave programmatically, so governance and design controls matter. Always reinforce local data governance and acceptable-use policies before anyone uses sensitive information.
