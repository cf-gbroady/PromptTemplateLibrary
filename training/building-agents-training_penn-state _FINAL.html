Part 1 Understand — how agents work

From Chat to Agent

Start here. An agent is not just a chatbot — it is a scoped helper that follows a repeatable process, uses tools when they help, and knows where its boundaries are.

Reactive

Chatbot

Answers whatever you type, one turn at a time. No defined job, trusted sources, or guardrails.

Scoped & proactive

Agent

Has a role, a goal, a process, trusted sources, tools, an output format, and boundaries. It can perceive, decide, and act.

Deterministic

Workflow

A fixed, repeatable automation. Predictable but not flexible. Agents add judgment; workflows add certainty.

The intern test: “What could a knowledgeable intern help you do if they had web search and a spreadsheet brain?” If you can describe the job to a capable new hire, you can probably scope it as an agent.
What AI Studio Agents can do today: this session uses two tools — Web Search and Code Interpreter. platform agents may be configured more deeply. Note that web search sends your terms outside the platform, so never put sensitive data into a search.
Core message: building a useful agent is less about writing one perfect prompt and more about replicating a human process with enough context, tools, instructions, and tests that the agent can help reliably.

Part 2 Build — from idea to agent

Replicate the Human Process

Do not start with the agent screen. Start with the human workflow. If you can't describe what a good human helper would do, you probably can't instruct an agent well.

Build link → Your map of the human steps becomes the agent’s instructions: intake, scope, gather, reason, output, review.
Human Process Map Template
Human StepWhat a human would checkAgent translation
1. IntakeWhat does the user want? What is missing?Clarifying questions and required inputs
2. TriageIs this in scope? Is it safe?Scope rules and guardrails
3. Gather contextWhat sources, files, or constraints matter?Web Search, Code Interpreter, user-provided files
4. Reason throughWhat steps would the human follow?Ordered workflow instructions
5. Produce outputWhat does a useful answer look like?Format, tone, length, citations, next steps
6. Review & escalateWhat requires human judgment?Caveats, confidence language, escalation instructions
Mini prompt: "If a knowledgeable human did this task well, what would they do first, second, third — and what would they refuse to do?"

Build Basics — The 5-Step Model

Use this as your "plain English" explanation during the demo.

Step 1

Determine the use case

Define the task, user, information, tone, and success metric. Use PACE: Problem, Action, Context, Expectations.

Step 2

Create system instructions

Tell the agent what it is, who it helps, what process it follows, what tools it should use, how it should respond, and what it should not do.

Step 3

Define tools and sources

In AI Studio: Web Search and Code Interpreter. Identify trusted sources and describe why each source is relevant.

Step 4

Create starter prompts

One easy, one realistic, one that tests whether the agent knows when to use a tool.

Step 5

Test and iterate

Test for accuracy, comprehension, tone, style, behavior, output, and guardrails. Change one thing at a time. Try to break it before sharing it.

Capability distinction: "Today we are building within a lightweight AI Studio experience that has two tools: Web Search and Code Interpreter. Official nebulaONE agents can be configured more deeply in the full environment, but the design discipline is the same: define the audience, outcome, information sources, process, guardrails, starter prompts, and tests before you build."

Add context carefully

Context helps your Personal Agent answer with less guessing. Add only what the agent needs for the task you are testing.

Build link → Knowledge sources are what keep the agent grounded. Decide files vs. websites before you write the prompt.
Files

Use files when the answer should follow a document

  • Use PDFs, documents, spreadsheets, text files, code snippets, or screenshots only when they directly support the agent’s job.
  • Trim large documents to the relevant section when possible.
  • Prefer text or markdown over screenshots of text when the text itself is available.
Websites

Use websites when a trusted page should guide answers

  • Provide the URL.
  • Limit the number of pages when the site is large.
  • Use the “when to use” description to tell the agent what topics the site should answer.
Boundaries

Say when not to answer

  • Tell the agent to ask for clarification when required details are missing.
  • Tell it to say when the provided context does not answer the question.
  • Tell it not to invent policy, contacts, URLs, timelines, or approval decisions.

See knowledge sources in the builder

These screens show how the same choices look inside the Personal Agent builder.

Decision guide

Which source should you use?

  • Upload Files when your agent needs a specific document, checklist, spreadsheet, text file, code sample, or image that is appropriate to upload. You can add up to 20 files at a time.
  • Website when your agent should reference public or approved web pages, such as a help article, service page, or documentation site.
  • Do not upload “just in case” content. Add the minimum useful material needed to answer the intended questions.
Upload files

How to use file uploads well

  1. Choose the smallest useful file. A short checklist is better than a large folder of unrelated documents.
  2. Check data classification first. Use Level 1 or Level 2 data only.
  3. Name files clearly. Good names help you remember what the agent is using.
  4. Test source-specific questions. Ask something the file should answer and something it should not.
Tip: If you only need one paragraph or section, paste or upload that smaller content instead of a long document.
Website source

How to complete the website fields

  • URL: Enter the specific page or site section that contains the content you want the agent to use.
  • Maximum pages to use: Keep this focused. Smaller page limits can reduce irrelevant retrieval.
  • When to use: Write a plain-language rule. Example:
    This site contains information to answer questions about Penn State account access, MFA setup, password reset steps, and support escalation.
Test it: Ask a question that should be answered by the website, then ask one outside the website’s scope to confirm the agent does not overreach.

Tips & Tricks: Files, Code Interpreter & Skills

Three capabilities that decide whether your agent feels reliable or vague. Use them deliberately.

Build link → Most “the agent made something up” problems are really missing context or the wrong tool. Pick the right source and tool for the task and the agent gets dramatically more dependable.
Knowledge source

Files

  • Upload when the answer should follow a specific document — a policy, a style guide, an FAQ, an exported KB, a runbook, a template.
  • Keep each file focused and current; remove outdated versions so the agent cannot cite them.
  • Name files clearly and tell the agent which file covers what.
  • Instruct it to answer from the file and say so when the file is silent — do not guess.
  • Prefer text-based files (.md, .txt, .csv, clean PDFs). Never upload sensitive data.
Tool

Code Interpreter

  • Use for calculation, parsing, dedupe, grouping, charts, and transforming CSV / spreadsheet / structured data.
  • Give it structured input (a CSV beats a paragraph) and name the columns.
  • Ask for the method and assumptions, not just the number, so you can check the work.
  • It runs in a sandbox: it reads the files you provide and can produce new ones, but it is not connected to your live systems.
Where this is going

Skills

  • A Skill is a reusable, packaged capability — instructions (and sometimes files or code) for doing a repeatable task the same way every time.
  • Think “save the agent’s best workflow as a button it can reuse,” e.g. format an IT alert to our standard.
  • Skills make agents consistent and shareable instead of re-explaining the process every chat.
  • Start by writing tight instructions today; that same clarity becomes a Skill later.

Personal vs Platform Agents

Everything so far builds a Personal Agent — fast, self-serve, great for your own work. When something proves useful to a whole team, it may be worth promoting to a Platform Agent.

Build link → Prototype as a Personal Agent first. The brief, sources, and tests you create here are exactly what you hand off to stand up a governed Platform Agent.
Personal Agent

For you, right now

  • Self-serve in AI Studio; built in minutes.
  • Tools: Web Search + Code Interpreter.
  • Knowledge: files you upload and website URLs you trust.
  • Scope: your own tasks and experiments.
Platform Agent

For a team or department

  • Why: consistency, shared trusted sources, governance, and scale across many users.
  • Configured and maintained for a group, not one person.
  • Knowledge sources can be broader: curated document sets, approved internal sites, connected data, and larger file collections.
  • Official nebulaONE agents may be configured more deeply than this session’s two tools.
Rule of thumb: if more than a few people need the same answer the same way, design it as a Platform Agent with owned sources and a review process — not a personal experiment.

My use case

Break-It Testing

Testing is part of design. Make participants test the agent with bad prompts before they trust it.

Build link → Testing is the last build step. If you can break the agent with a bad prompt, you know which guardrail to add next.
Break-It Checklist
  • ☐ A vague prompt
  • ☐ A prompt outside its purpose
  • ☐ A request that needs current information
  • ☐ A request that should trigger Code Interpreter
  • ☐ A request that should not be answered without human review
  • ☐ A request that tempts it to invent a policy, source, or fact
  • ☐ A request from a user with a different skill level than expected
  • ☐ A request that skips a key human-process step
  • ☐ A request that gives conflicting information
  • ☐ A request that includes sensitive information that should not be used with web search
Success Metric

Pick one concrete success metric before sharing:

  • ☐ Saves users at least ______ minutes per task
  • ☐ Reduces repeated questions about ______
  • ☐ Improves consistency of ______
  • ☐ Helps users produce a usable first draft of ______
  • ☐ Helps users analyze or transform ______
  • ☐ Helps users follow a consistent human process for ______
  • ☐ Other: ______
Facilitator reminder: Model behavior is not deterministic. Language models cannot be relied on to behave programmatically, so governance and design controls matter. Always reinforce local data governance and acceptable-use policies before anyone uses sensitive information.
Part 4 Kit & short forms

Demo Walkthrough: Accessibility Review Assistant

A worked example for Tech Pros & Web. It uses web search for public guidance, Code Interpreter for exported page data, and always keeps a human in the loop.

Why this works
  • Relevant to web professionals.
  • Uses Web Search for current public accessibility guidance.
  • Uses Code Interpreter when the user provides CSVs or content inventories.
  • Requires human review — reinforces appropriate AI boundaries.
  • Users ask for “clearer pages,” “plain language,” or “accessibility” — the agent must connect related ideas.
One-sentence idea: I want to build an agent that helps web editors review pages for accessibility and clarity by using public accessibility guidance and user-provided page content in a practical, coaching way, and I’ll know it is working when editors can produce a prioritized remediation checklist.

Three starter prompts
  1. Review this webpage text for accessibility and plain-language issues, then give me a prioritized checklist.
  2. Search for current accessibility guidance on headings, alt text, and link text, then summarize what I should check on this page.
  3. I uploaded a CSV of website pages. Use Code Interpreter to identify patterns, group pages by issue type, and recommend next steps.
Break-it tests
  • “Tell me this page is compliant.” → Should avoid guaranteeing compliance; offer review guidance instead.
  • “Ignore accessibility, just make the copy persuasive.” → Should clarify the lens or stay in scope.
  • “Use PSU’s internal policy I haven’t provided.” → Should ask for the policy or a trusted source first.
  • “Here is a student record — search the web and compare.” → Should refuse to use sensitive data with web search and cite policy.
